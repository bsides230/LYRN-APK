How the Model Actually Runs Right Now

This page explains the current live LYRN execution path: how the system maintains continuity via the Relational Web Index (RWI), how it leverages the KV Cache for performance, and how temporary job instructions are loaded and unloaded cleanly using Dynamic Snapshots (via DSManager).

Cognition Memory-Based
KV Cache Persistent
Instructions Dynamic Snapshots
Output Separated
Core Principle
LYRN moves beyond prompt injection. It loads structured memory into RAM (the Master Snapshot) and reuses it across turns via the KV Cache. Dynamic job logic is injected cleanly as temporary snapshots and drops away when the job is done, preventing prompt drift.
Execution Shape
watch → build → run → capture

The runner stays idle until triggered, assembles the active context stack, generates, writes raw output, and then hands the result to the capture layer.

KV Cache Advantage

By keeping the static Master Snapshot loaded once and reusing its tokens, the model maintains consistent cognitive behavior without token churn or re-evaluating identity at every step.

Job Scoping

Job instructions are now pure Dynamic Snapshots. DSManager activates them early in the context, and turns them off when the job completes, preventing stale instructions from polluting future reasoning.

02 // Injection Order Inside the Model Context
01

Master Prompt / RWI (Relational Web Index)

Not just memory, and not just a prompt. The RWI is a structured relational cognition space. It forms the environment the model operates inside. Because it sits at the bottom of the stack, it stays locked in the model's KV Cache across turns.

Purpose: Establish the persistent system frame without re-evaluating core identity on every turn.
02

Active Dynamic Snapshots (via DSManager)

This layer holds the active Job Instructions. When a job runs, its instructions are activated as a dynamic snapshot. By injecting it early (right after the master prompt), it benefits from KV Cache reuse during the job, but cleanly unloads when the job finishes.

Snapshots exist in DSManager as registered objects. Only active snapshots are injected into the model context, and this injection happens only during context assembly before generation. Snapshots are NOT removed mid-run. After the job completes, the snapshot is excluded from the next context build, ensuring no stale instructions persist across jobs.

Purpose: Inject bounded, temporary job logic without cluttering the bottom of the prompt or impersonating the user.
03

Chat History

The runner adds prior user/assistant turns. This preserves the conversational thread, giving the model continuity without confusing history with system-level control layers.

Purpose: Preserve conversational memory and turn-by-turn context.
04

Deltas / Mutable Runtime State

Deltas are injected fresh every run and are NOT relied on for KV cache reuse. They represent fast-changing runtime state (like current time or hardware stats) and do NOT define reasoning structure. Unlike Snapshots (which persist for a job, affect reasoning, and form the structured cognitive frame), Deltas merely provide immediate environmental awareness.

Purpose: Inject fast-changing state without rewriting the stable KV-cached layers.
05

Actual User Message

The final layer is the real user text as the true user turn. Everything before it is preparation, framing, or runtime context.

Purpose: Ensure the model responds directly to the person/input, not to internal orchestration text.
03 // What Each Layer Is Doing
A

RWI / Master Snapshot

Not just memory or a prompt—it is the structured relational cognition space that forms the environment the model continuously operates inside.

B

Dynamic Snapshots (Job Logic)

Job instructions that persist for the duration of a job. They become part of the structured cognitive frame and actively affect reasoning behavior and task execution.

C

Conversation History

Recent user/assistant turns. Gives the model continuity and the ability to answer in the context of an ongoing exchange.

D

Delta Layer

Fast-changing runtime facts injected fresh every run. They do NOT define reasoning structure and are NOT relied on for KV cache reuse.

E

User Turn

The pure, unadulterated user input. Never mixed with system instructions.

F

KV Cache Reuse

KV cache reuse depends entirely on prefix stability. The Master Snapshot acts as the long-lived cache layer, while Dynamic Snapshots form the job-scoped cache layer. If a Dynamic Snapshot changes, cache reuse stops at that exact point, and everything after that layer is recomputed.

04 // Why the Order Matters
KV Cache Efficiency

Identity before state

The stack moves from the most stable layers to the most volatile. First comes enduring system identity (RWI), then job framing (Dynamic Snapshots), then history, then live overlays (Deltas), and finally the user turn.

This allows the model to cache the early parts of the prompt. If the system identity and job instructions haven't changed since the last turn, the LLM skips parsing them, saving thousands of tokens per run.

Job Boundedness

Instructions via DSManager

In the past, job instructions were appended late, polluting the user's turn. Now, Job Instructions are the Dynamic Snapshot. They are injected cleanly as a system layer.

When the job finishes, the Scheduler tells DSManager to deactivate the snapshot. The next job runs with a clean slate, preventing stale instructions from bleeding over.
